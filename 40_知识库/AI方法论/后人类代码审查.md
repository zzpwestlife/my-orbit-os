# 后人类代码审查

## 定义

在 Vibe Coding 时代，放弃逐行人工 review 代码，转而采用 **Agent Review + E2E Test** 的自动化审查模式。承认人类已无法有效审查 AI 生成的大规模代码。

## 核心观点

> "逐行 review 是时代糟粕。Agent review + e2e test 才是在 vibe coding 时代的提效工具。"

### 为什么人类 Review 失效了

| 问题 | 说明 |
|-----|------|
| **代码量爆炸** | AI 生成速度远超人类阅读速度 |
| **非自写代码** | 大部分代码不是你写的，理解成本高 |
| **表面审查** | 人类 review 往往只能发现风格问题，深层逻辑 bug 难以察觉 |
| **时间成本** | 逐行 review 抵消了 AI 的提效收益 |

## 两派观点

### 保守派：Agent Review + E2E Test

```
AI 写代码 → Agent Review → E2E Test → Merge
```

- 用另一个 agent（或同一 agent 的第二次调用）进行代码审查
- 依赖端到端测试验证功能正确性
- 人类只关注架构设计和需求对齐

### 激进派：自我 Review 后直接 Merge

```
AI 写代码 → Prompt: "深度 review 代码" → Merge
```

> "Agent 代码质量很高了，在他写完后，再 prompt 一次 '深度 review 一下你的代码，确保高质量、高鲁棒性、高可用性、不破坏过去的行为、不引入 bug'，然后就 merge。"

**前提假设：**
- 代码由高质量模型（Sonnet/Opus/GPT-4）生成
- 项目有完善的自动化测试覆盖
- 可以快速回滚

## 实践建议

### 何时使用激进派方案

✅ 适合激进派：
- 独立功能模块
- 有完整测试覆盖
- 非核心敏感代码
- 快速迭代原型

⚠️ 建议保守派：
- 核心支付/安全模块
- 数据库 schema 变更
- 缺乏测试的老项目
- 合规审计严格的环境

### Agent Review Prompt 模板

```
请深度 review 这段代码，检查：
1. 是否破坏向后兼容性
2. 是否有潜在的边界情况未处理
3. 是否有性能隐患
4. 是否符合项目代码规范
5. 是否有安全风险

输出：问题列表 + 修复建议（如有）
```

## 风险与缓解

| 风险 | 缓解措施 |
|-----|---------|
| Agent 自我审查盲区 | 使用不同模型或多次独立审查 |
| 测试覆盖不足 | 强制要求新代码必须有测试 |
| 架构腐化 | 定期人类参与架构 review |
| 回滚困难 | 小批量、频繁提交 |
